
#Cross-Reference QA Benchmark: PageIndex vs FAISS

Benchmarks vectorless RAG (PageIndex) against vector RAG (FAISS) on cross-reference
questions across NumPy and Pandas documentation.

 #Key Finding

On multi-hop questions where PageIndex successfully navigated to the target page,
it achieved **90.4% accuracy vs 75.0% for FAISS (+15.4%)**. Even when navigation
failed, PageIndex outperformed FAISS **87.5% vs 62.5%** — demonstrating that
tree-based document navigation is structurally superior to vector similarity for
cross-reference QA.

| Metric | FAISS | PageIndex |
|---|---|---|
| Overall accuracy | 79.8% | **88.4%** |
| Single-hop accuracy | 85.5% | 87.0% |
| Multi-hop accuracy | 73.3% | **90.0%** |
| Multi-hop (navigation succeeded) | 75.0% | **90.4%** |
| Multi-hop (navigation failed) | 62.5% | **87.5%** |
| Avg latency | 3.0s | 43.0s |
| Avg input tokens | 2,212 | 10,617 |

## Why this matters

When developers ask cross-reference questions like *"How does np.einsum interact
with broadcasting rules?"*, the answer requires visiting multiple doc sections.
Vector RAG retrieves the einsum page and stops — it never follows the "see also:
numpy.broadcast" link. The answer is incomplete.

PageIndex navigates the doc hierarchy like a human — reads the einsum section,
sees the cross-reference, follows it, retrieves the broadcasting section, and
synthesizes a complete answer. This is a structural failure of vector RAG, not
a tuning problem. No amount of better chunking or embeddings fixes it.

## What gets built

Two complete retrieval pipelines evaluated against a 129-question benchmark:

**Pipeline A — FAISS (Vector RAG)**
- Chunks NumPy/Pandas docs into 300-word segments
- Embeds with `all-MiniLM-L6-v2`
- Retrieves top-5 chunks by cosine similarity
- Answers via Gemini 2.5 Flash

**Pipeline B — PageIndex (Vectorless RAG)**
- Builds a hierarchical tree from the doc structure
- LLM navigates the tree top-down, following cross-reference links
- No embeddings — pure LLM reasoning over document structure
- Answers via Cerebras gpt-oss-120b

## Dataset

129 questions across NumPy and Pandas documentation, each requiring at least one
explicit cross-reference to answer correctly.

| Split | Count |
|---|---|
| Total questions | 129 |
| NumPy | 64 (49%) |
| Pandas | 65 (51%) |
| Single-hop | 69 (53%) |
| Multi-hop | 60 (47%) |

Questions were generated by prompting Gemini to produce questions that are
unanswerable from the source page alone. Ground truth answers were pulled directly
from the referenced target pages. 50 questions were manually verified.

## Stack

- `scrape.py` — BFS crawler for NumPy + Pandas docs
- `parse.py` — cross-reference pair extractor
- `generate.py` — Gemini-based question generator
- `package.py` — dataset packager
- `pipeline_faiss.py` — FAISS retrieval pipeline
- `pipeline_pageindex.py` — tree-based vectorless RAG pipeline
- `evaluate.py` — LLM-as-judge evaluation harness (Cerebras)

## Setup

```bash
pip install -r requirements.txt
cp .env.example .env

```

## Run the full pipeline

```bash
# Stage 1-4: build the dataset
python scrape.py        # crawl NumPy + Pandas docs (~15 min)
python parse.py         # extract cross-reference pairs (~1 min)
python generate.py      # generate questions via Gemini (~15 min)
python package.py       # package final dataset (~5 sec)

# Stage 5-6: run retrieval pipelines
python pipeline_faiss.py      # vector RAG (~7 min)
python pipeline_pageindex.py  # vectorless RAG (~90 min)

# Stage 7: evaluate
python evaluate.py      # LLM-as-judge scoring (~15 min)
```

## Difficulty levels

- `single_hop` — answer requires following ONE cross-reference (source → target)
- `multi_hop` — answer requires following TWO OR MORE cross-references

## Navigation success rate

PageIndex successfully navigated to the target page on **79.8%** of questions.
Accuracy when navigation succeeded: **92.2%**. Accuracy when navigation failed:
**73.1%** — still above FAISS overall (79.8%), suggesting the tree structure
provides value even when the exact target page is missed.

## Limitations

- PageIndex is 14x slower (43s vs 3s avg latency) and uses 5x more tokens
- For single-hop questions the accuracy gap narrows to 1.5% — FAISS is more
  cost-efficient there
- Optimal production architecture: route simple queries to FAISS, complex
  cross-reference queries to PageIndex

## Manual verification

Open `output/verification_sample.json` and for each question:
1. Read the question
2. Open `source_url` — confirm you **cannot** answer from this page alone
3. Open `target_url` — confirm the `ground_truth_answer` is accurate
4. Check the `difficulty` label is correct
5. Mark PASS / FAIL

Aim for >85% pass rate before using the dataset in evaluations.
